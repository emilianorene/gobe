# NES / Famicom Disk System (FDS) Support Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the user's 194 `.fds` (Famicom Disk System) games playable under `System.NES` by bundling/pinning an FDS-capable Libretro core and wiring `System.NES` into `CoreManager`.

**Architecture:** Reuse the Fase 2 + Arcade stack unchanged (`EmulatorActivity` + `GLRetroView`, `CoreManager`, jniLibs core packaging, external `systemDirectory` for BIOS, `getGLRetroErrors` safety net, `playable = corePath != null`). Spike an FDS core on-device against a real `.fds` (with the user's `disksys.rom` in place), pin it, bundle if needed, and map `System.NES → core`.

**Tech Stack:** Kotlin, Jetpack Compose for TV, LibretroDroid 0.14.0, an FDS-capable Libretro core (FCEUmm candidate, `armeabi-v7a`), Room, JUnit (JVM) + on-device verification over wireless adb.

**Spec:** `docs/superpowers/specs/2026-06-30-gobe-fds-design.md`

## Notes for the implementer

- **BIOS is the #1 trap:** run the spike ONLY with `disksys.rom` in `/Download/ROMs/system`, else
  every core looks broken.
- **ABI:** any new core `.so` must be `armeabi-v7a` (32-bit) — verify with `file`.
- **Single source of truth:** `CoreManager` decides playability; detail + `EmulatorActivity`
  already consult it — no changes needed there.
- **Don't bundle BIOS** or ROMs. **Fail fast on FBNeo** for FDS; FCEUmm is the likely pin.
- **Don't fix romsets / disk images:** surface failures, don't repair the user's `.fds`.

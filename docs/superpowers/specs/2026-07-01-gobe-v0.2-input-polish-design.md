# Gobe v0.2 — Input & In-Game Polish

> Date: 2026-07-01
> Scope: Four small, cohesive input/in-game features for the v0.2 milestone — unassign a
> controller, FDS disk-swap from the pause menu, an analog stick deadzone setting, and a
> configurable menu hotkey. Follows the established pattern: pure testable logic in
> `emulation/input/`, persistence in the shared `gobe.settings` prefs, thin Activity/Compose wiring.

## 1. Purpose

Close gaps found while using the v0.1.0-beta and round out controller/in-game handling:

- **Unassign a controller** — today a pad can be set to P1–P4 but there is no way to clear it back to
  unassigned.
- **FDS disk-swap** — multi-disk Famicom Disk System games (e.g. Zelda, Metroid) need a manual disk
  flip; there is no UI for it.
- **Analog deadzone** — worn sticks can drift and inject phantom input; there is no way to filter it.
- **Configurable menu hotkey** — the pause menu combo is hard-coded to Select+Start; some controllers
  or preferences want a different combo.

## 2. Decisions captured during brainstorming

- **One spec, four bounded units.** All four are small and share the input/in-game theme. Each gets
  its own pure module (where applicable) + a thin wiring change; each is independently testable.
- **Deadzone default = Low (0.10).** A mild deadzone that helps drift without hurting normal play.
  Levels are discrete: Off / Low / Med / High.
- **Disk-swap cycles to the next disk** (wrapping), rather than a "choose disk N" picker — simplest
  and sufficient for FDS (usually 2 sides).
- **Menu hotkey is the simultaneous combo only.** Back (remote) and the gamepad Home/Mode button
  always toggle the menu regardless of the setting; only the configurable two-button combo is
  swappable. Individual buttons still reach the core (so the combo's buttons stay usable in games).
- **Read-at-launch for game settings.** Deadzone and menu-hotkey are read in `EmulatorActivity.onCreate`
  (like swaps/remaps); changing them mid-session needs a relaunch. Documented.

## 3. In scope

### 3.1 Unassign a controller
- No new model/persistence: `ControllerAssignments.clear(descriptor)` already exists, and
  `ControllerPrefs.save` rewrites all `ctrl.port.*` keys from the map, so clearing + saving removes
  the key.
- `ControllerDetailScreen`: add an **"Unassign"** button to the player-selector `Row` after P1–P4,
  marked selected (●) when `assignedPort == null`. New callback `onUnassign: () -> Unit`.
- `ControllersActivity`: `onUnassign = { assignments = assignments.clear(sel); ControllerPrefs.save(this, assignments) }`.
- Reuse the existing `controllers_unassigned` string ("Not assigned" / "Sin asignar").

### 3.2 FDS disk-swap
- **Pure** (`emulation/input/DiskControl.kt`): `fun nextDisk(current: Int, count: Int): Int` =
  `if (count <= 1) current else (current + 1) % count` (also treats a negative/zero count safely).
- `PauseOverlay`: new params `diskCount: Int`, `currentDisk: Int`, `onChangeDisk: () -> Unit`. Render a
  **"Change disk (n/N)"** button (`currentDisk + 1` / `diskCount`) placed above the legend, **only when
  `diskCount > 1`**.
- `EmulatorActivity`: `diskCount`/`currentDisk` as `mutableStateOf` ints (default 0/0). Populate from
  `retroView.getAvailableDisks()` / `getCurrentDisk()` in `onCoreReady()`. `onChangeDisk()` computes
  `nextDisk`, calls `retroView.changeDisk(next)`, updates `currentDisk`, and toasts the new disk
  number. Non-FDS games report `availableDisks <= 1` → the button is hidden.

### 3.3 Analog deadzone
- **Pure** (`emulation/input/Deadzone.kt`):
  - `enum class DeadzoneLevel(val threshold: Float) { OFF(0f), LOW(0.10f), MEDIUM(0.15f), HIGH(0.25f) }`.
  - `fun applyDeadzone(value: Float, threshold: Float): Float` — per-axis with rescale:
    if `threshold <= 0f` return `value`; if `abs(value) < threshold` return `0f`; else rescale the
    remaining range so output starts at 0 at the threshold edge and reaches ±1 at ±1:
    `sign(value) * ((abs(value) - threshold) / (1f - threshold))`, clamped to [-1, 1].
- `EmulatorActivity.onGenericMotionEvent`: apply `applyDeadzone(_, level.threshold)` to the analog
  axes only — `AXIS_X/AXIS_Y` (left) and `AXIS_Z/AXIS_RZ` (right). **Do not** apply it to `AXIS_HAT_X/Y`
  (the D-pad is digital). Load the level in `onCreate`.
- `SettingsScreen`: a "Deadzone" row with four buttons (Off/Low/Med/High) mirroring the language
  selector pattern; the active level shows ●. Persist via `GameSettings`.

### 3.4 Configurable menu hotkey
- **Pure** (`emulation/input/MenuHotkey.kt`):
  - `enum class MenuHotkey(val codes: Set<Int>) { SELECT_START(setOf(KEYCODE_BUTTON_SELECT, KEYCODE_BUTTON_START)), L1_R1(setOf(KEYCODE_BUTTON_L1, KEYCODE_BUTTON_R1)), L3_R3(setOf(KEYCODE_BUTTON_THUMBL, KEYCODE_BUTTON_THUMBR)) }`.
  - `fun isHotkeyCombo(held: Set<Int>, hotkey: MenuHotkey): Boolean = hotkey.codes.all { it in held }`.
- `EmulatorActivity.dispatchKeyEvent` generalizes the current Select+Start special-case:
  - Keep the **Back** and **Button MODE** branches unchanged (always toggle).
  - For any keycode in the configured `hotkey.codes`: track it in `heldKeys` on DOWN/UP, **forward it
    to the core as normal input** (so the buttons stay usable in games), and when
    `isHotkeyCombo(heldKeys, hotkey)` is true and `!paused`, clear `heldKeys`, toggle, and swallow.
    While paused, defer to `super` (overlay nav), same as today.
  - Keys not in the hotkey set follow the existing normal-forward path.
- `SettingsScreen`: a "Menu hotkey" row with three buttons (Select+Start / L1+R1 / L3+R3), active one
  shows ●. Persist via `GameSettings`.
- **Legend text follows the setting:** `ControlsHint` (launch hint) and the `PauseOverlay` legend show
  the configured combo's label (e.g. "L1 + R1") instead of a hard-coded "Select + Start". A small
  `MenuHotkey.label` (or a `@StringRes` mapping) provides the display string; Back is always appended.

### 3.5 Shared persistence — `GameSettings`
- New `emulation/GameSettings.kt` (object, same shape as `ControllerPrefs`), reading/writing the
  shared `gobe.settings` prefs:
  - `loadDeadzone(context): DeadzoneLevel` / `saveDeadzone(context, level)` — key `game.deadzone`
    stored as the enum name; unknown/missing → default **LOW**.
  - `loadMenuHotkey(context): MenuHotkey` / `saveMenuHotkey(context, hotkey)` — key `game.menuhotkey`
    stored as the enum name; unknown/missing → default **SELECT_START**.

## 4. Out of scope (deferred)

- Radial (magnitude-based) deadzone — per-axis is sufficient for these systems; documented.
- Per-game settings (global only for now).
- Disk picker ("choose disk N") and disk-side labels — cycle is enough.
- Single-button or triple-button menu hotkeys; rebinding Back/Mode.
- Rumble (infeasible on this hardware), USB host (won't-fix hardware).

## 5. Architecture

- **Pure modules (JVM-testable), no Android deps beyond `KeyEvent`/`MotionEvent` constants:**
  - `DiskControl.nextDisk(current, count)`.
  - `Deadzone`: `DeadzoneLevel` enum + `applyDeadzone(value, threshold)`.
  - `MenuHotkey`: enum + `isHotkeyCombo(held, hotkey)` + a display label.
- **Persistence:** `GameSettings` (object) over `gobe.settings`, storing enum names; defensive
  parsing with defaults (LOW, SELECT_START).
- **Emulator wiring (`EmulatorActivity`):** load `deadzone` + `menuHotkey` in `onCreate`; disk state in
  `onCoreReady`; apply deadzone in `onGenericMotionEvent`; generalize the combo in `dispatchKeyEvent`;
  pass disk state + `onChangeDisk` into `PauseOverlay`.
- **UI:** `PauseOverlay` (disk button, dynamic legend), `ControllerDetailScreen` (Unassign button),
  `SettingsScreen` (Deadzone + Menu hotkey rows), `ControlsHint` (dynamic combo text), `strings.xml`.

## 6. Data flow

1. **Settings screen** writes `game.deadzone` / `game.menuhotkey` via `GameSettings`.
2. **Controllers screen** writes/clears `ctrl.port.<desc>` via `ControllerPrefs` (unassign path).
3. **Launch:** `EmulatorActivity.onCreate` reads deadzone + menu-hotkey once; `onCoreReady` reads disk
   count/current.
4. **In-game:** motion events are deadzoned per-axis before `sendMotionEvent`; key events check the
   configured combo; the pause overlay's disk button calls `changeDisk` and refreshes `currentDisk`.

## 7. Error handling / edge cases

- `nextDisk` with `count <= 1` returns `current` (no-op); the button is hidden anyway.
- `getAvailableDisks`/`getCurrentDisk`/`changeDisk` wrapped so a core that doesn't support disks can't
  crash the overlay (default 0 disks → hidden button).
- `applyDeadzone` with `threshold <= 0` (OFF) is a passthrough; clamp output to [-1, 1].
- `GameSettings` parse of an unknown enum name falls back to the default, never throws.
- Unassign when already unassigned is a harmless no-op (clear on an absent key).

## 8. Testing

- **Unit (JVM):**
  - `nextDisk`: wraps `count-1 → 0`; mid-range increments; `count <= 1` and negative counts return
    `current`.
  - `applyDeadzone`: below threshold → 0; at/above → rescaled (0 at the edge, ±1 at ±1); OFF
    (threshold 0) passthrough; clamps and preserves sign; symmetric for negatives.
  - `isHotkeyCombo`: true only when all of the hotkey's codes are held; false for partial/empty; each
    enum value matches its own codes and not another's.
  - `ControllerAssignments.clear` already covered; add a case asserting `portFor` is null after clear.
- **On-device (controller awake):**
  - **Unassign:** assign a pad to P2, tap Unassign → detail + list show "Not assigned"; launch a game
    → the pad drives P1 (fallback).
  - **Disk-swap:** launch a multi-disk FDS game → pause → "Change disk (1/2)" visible → change →
    toast "Disk 2" and the game reads the other side; a single-disk/non-FDS game shows **no** disk
    button.
  - **Deadzone:** set High → normal stick/D-pad input still works, small idle drift is ignored; Off
    restores raw behavior.
  - **Menu hotkey:** set L1+R1 → L1+R1 opens the menu, Select/Start now pass through to the game;
    Back and Home/Mode still open the menu; the launch hint + pause legend read "L1 + R1".

## 9. Risks

1. **`changeDisk` semantics** — LibretroDroid's `changeDisk` must perform the eject→set-index→insert
   cycle internally. If a core needs an explicit eject step, the swap may not register. Mitigation:
   verify on-device with a real multi-disk FDS game before merge; if it fails, investigate the AAR's
   disk-control path. This is the one item that can't be fully proven by unit tests.
2. **Per-axis vs radial deadzone** — per-axis yields a square deadzone; acceptable for D-pad-era
   games and documented. Low value overall (these cores are mostly digital).
3. **Combo forwarding** — the hotkey's buttons are forwarded to the core until the combo completes
   (same as today's Select+Start). A brief press of one hotkey button reaches the game; acceptable and
   pre-existing behavior.
4. **Read-at-launch** — deadzone/menu-hotkey changes need a game relaunch to take effect (consistent
   with swaps/remaps). Documented; not surfaced in UI.

## 10. Defaults (change on request)

- Deadzone: **LOW (0.10)**; levels Off/Low/Med/High.
- Menu hotkey: **SELECT_START**; options add L1+R1 and L3+R3; Back + Home/Mode always toggle.
- Disk-swap: cycle to next disk (wrapping); button only when >1 disk.
- All settings global and read-at-launch; enum names persisted in `gobe.settings`.
